DocManager

A full-featured document management system built with Laravel 12. Supports file upload, content extraction, TF-IDF semantic search, role-based access control, real-time chat, developer assessments, and a REST API — all served from a single application.

What it does

📄

Document Store

Upload, version, categorise and tag documents of any type.

🔍

Dual Search

Keyword (MySQL FULLTEXT) and TF-IDF semantic search over extracted content.

👥

Role-based Access

Admin, Editor, and Viewer roles with per-user storage quotas.

🔗

Share Links

Time-limited public share tokens. No login required for recipients.

🔒

Lock & Archive

Admins can lock documents to block downloads/previews, or auto-archive on a date.

🌐

REST API

Sanctum token-based API for programmatic document access and search.

💬

Real-Time Chat

1:1 messaging over a self-hosted WebSocket server (Laravel Reverb) — no third-party broadcaster.

🧪

Developer Tests

Build a timed coding test from the problem bank, send a no-account invite link, grade manually.

👤

HR & People

Employee profiles, departments, positions, documents, and manager hierarchy.

🕐

Attendance

Daily check-in/out records, work hours, and leave requests with approval workflow.

📋

Work Reports

Daily/weekly/monthly reports with task tracking, manager review, and analytics.

🗂️

Plans

Full plan lifecycle with task lists, employee assignment, progress tracking, and comments.

Quick Start

From clone to running server in four steps.

One-command setup

composer run setup

This single script runs: composer install, copies .env.example → .env, generates the app key, runs migrations, npm install, and npm run build.

Start the dev stack

composer run dev

Starts four concurrent processes via concurrently:

ProcessCommandPurpose
serverphp artisan serveLaravel dev server on port 8000
queuephp artisan queue:listenProcesses ExtractDocumentContent & IndexDocumentTfidf jobs
vitenpm run devVite HMR for CSS/JS assets
reverbphp artisan reverb:startSelf-hosted WebSocket server for the chat feature's realtime message push

app.php

The central wiring file. Registers the three global middleware groups:

  • web group — sessions, CSRF, cookies, Authenticate redirect
  • api group — stateless Sanctum auth, throttle
  • Global aliases — role → RequireRole, active → RequireActiveAccount

Document Management

The core feature. Documents (called Resources in the codebase) move through a lifecycle from upload to archive.

Lock feature

When a document is locked (locked_by and locked_at are set):

  • download route returns a redirect-back with an error flash
  • stream route returns 403
  • preview route returns 403
  • The document detail page shows a yellow "Document locked" banner
  • The download button is replaced with a greyed-out "Locked" badge
  • The "Open Preview" button is hidden

Soft delete & trash

Deleting a document from the admin panel performs a soft delete (deleted_at timestamp). The document remains in storage/app/private/resources/ and can be restored. Permanent deletion (force delete) removes the DB record and the file from disk. The dms:prune-trash command auto-force-deletes documents soft-deleted longer than TRASH_RETENTION_DAYS.

Versioning

Each upload of a new version creates a DocumentVersion record. The current version number is tracked on the Resource. Admins can roll back to any previous version via VersionController@restore, which copies the old version's file path back to the resource and bumps the version number.

Search System

Two search modes: fast keyword search via MySQL FULLTEXT index, and semantic TF-IDF similarity search.

Keyword search

  • Uses MySQL MATCH(title, description, content) AGAINST(? IN BOOLEAN MODE)
  • Fast, exact, supports boolean operators (+word -word "phrase")
  • Results sorted by MySQL relevance score
  • Works out of the box with no pre-processing

TF-IDF semantic search ("AI Search")

  1. Build phase — search:build-tfidf runs two passes over all documents:
    • Pass 1: compute document frequency (DF) for every unique term across the entire corpus
    • Build IDF dictionary: log((N+1)/(df+1)) + 1 (smooth IDF) → saved to JSON
    • Pass 2: compute TF-IDF vector for each document → stored in resource_embeddings
  2. Query phase — When a user submits an "AI Search":
    • Tokenize and vectorize the query using the same IDF dictionary
    • Compute cosine similarity between query vector and each stored embedding
    • Return top-N matches sorted by similarity score

Indexed text

Both search modes index the combination of title + description + content. Content is extracted from the file by ContentExtractorService and stored in resources.content.

Brute-force lockout

After multiple failed logins, the account's locked_until timestamp is set. Login attempts before that time are rejected regardless of credentials.

Storage quotas

Each user has a storage_quota_mb. User::storageUsedBytes() sums file_size for all documents uploaded by the user. User::wouldExceedQuota($newBytes) is checked before accepting a new upload.

Users can request a username change or account deletion via the profile page. These create AccountRequest records with status = 'pending'. Admins review them at /admin/account-requests and approve or reject.

File Upload

Two upload paths: standard single-request upload and chunked upload for large files.

Standard upload

Admin → POST /admin/documents → DocumentController@store:

  1. Validate (file required, MIME type, max size)
  2. Generate UUID filename and store at storage/app/private/resources/{uuid}.ext
  3. Create Resource record with file_hash (MD5) for deduplication
  4. Dispatch ExtractDocumentContent job to queue
  5. Dispatch IndexDocumentTfidf job after extraction completes

Chunked upload

For large files, the browser uploads in N slices:

  1. POST /admin/documents/upload/chunk — receives each binary slice, stores in storage/app/chunks/{upload_id}/
  2. POST /admin/documents/upload/assemble — concatenates all chunks in order, verifies final hash, moves assembled file to resources/{uuid}.ext, creates Resource record, dispatches extraction job

File deduplication

The MD5 hash (file_hash) is stored on every Resource. On upload, if a matching hash already exists, the admin can be warned. Existing content extraction is reused.

Sharing

Time-limited public share links that don't require the recipient to log in.

How it works

  1. Logged-in user clicks "Generate Share Link" on a document's detail page
  2. DocumentController@share creates a Share record:
    • token: 40-char random string
    • expires_at: now + SHARE_LINK_EXPIRY_HOURS
    • revoked_at: null (until manually revoked)
  3. The URL /share/{token} is shown to the user
  4. ShareController@show validates Share::isValid() and shows the document (read-only, no download unless unlocked)

Share validation

public function isValid(): bool
{
    return is_null($this->revoked_at) && $this->expires_at->isFuture();
}

Cleanup

The scheduled command dms:prune-shares (runs daily at 02:15) deletes Share records that expired more than 7 days ago.

Developer Tests

Turns the problem bank into a timed coding assessment that an external candidate can take with no DocManager account, via a single invite link.

Building a test

An admin/editor picks any mix of existing bank problems (via an "Add Existing Problem" search modal) and/or types brand-new problems inline. Each inline problem becomes a real Problem row (category defaults to "Custom") so it's reusable in future tests too — no separate "custom problem" schema. A test has a title, optional description (shown to the candidate), a time limit in minutes, and a status: draft (not invitable), active (invites can be created), or archived.

Inviting a candidate

From the test's detail page, entering just a name (email is optional) generates a 48-character random token and a public link: /test/{token}. The link is unguessable, scoped to that one invite, and rate-limited at the route level (60 req/min) since it requires no authentication.

Taking the test

The candidate sees a start screen (title, description, problem count, time limit), then a single page with all problems and a countdown timer (mm:ss under an hour, hh:mm:ss at or above it) that auto-submits when it hits zero. If the candidate abandons the tab, the next time anyone loads that link (including the admin checking status), the server notices the timer has expired and force-finalizes the submission with whatever was filled in — no orphaned "in progress forever" invites.

Grading

Each problem carries a point value (default 100, editable per problem in the test). The grading screen shows the candidate's submitted code next to the problem's reference solution_code, with a score input (clamped server-side to the problem's point value) and a free-text feedback field. Saving computes total_score / max_score and flips the invite to graded.

Chat & Messaging

Real-time 1:1 messaging between any two active users, built on Laravel Reverb — a self-hosted WebSocket server, so there's no third-party broadcasting service (Pusher, Ably) in the loop.

Why Reverb

Consistent with the rest of the project's offline-first, self-hosted design: Reverb ships as a first-party Laravel package and speaks the Pusher protocol, so the standard laravel-echo + pusher-js client libraries work unmodified — just pointed at a server you run yourself (php artisan reverb:start).

Real-time pipeline

EventBroadcast onTriggers when
MessageSentprivate-conversation.{id} and the recipient's private-App.Models.User.{id}A message is created — the second channel lets the nav badge/sidebar update even if that conversation isn't open
MessagesDeliveredprivate-conversation.{id}The recipient's client fetches the thread (or acks a live message) and pending messages get a delivered_at timestamp
MessagesReadprivate-conversation.{id}The recipient opens the thread — upserts their conversation_reads.last_read_at pointer

All three use ShouldBroadcastNow (synchronous, no queue worker needed for chat to feel instant) and are wrapped in try/catch at the call site — if Reverb is briefly down, the underlying save still succeeds and only the live push is skipped.

Read & delivered indicators

Status ticks are derived, not stored per message: a sent message you own shows "read" if its timestamp is at or before the other participant's last_read_at, "delivered" if delivered_at is set, otherwise "sent". Marking an entire thread read is one upsert on conversation_reads, not N message updates.

Presence (online/offline)

A presence channel (presence-online) tracks who's currently connected — joining/leaving fires purely over the socket, no database polling. users.last_seen_at is updated opportunistically (throttled to once per minute) by RequireActiveAccount middleware as a fallback "last seen" for when a user is offline.

Other features

  • Search — conversations are filtered by participant name/username or by message content, scoped to the caller's own conversations
  • Emoji picker — a static grid of common emoji, no external package
  • Responsive layout — the conversation list and the open thread collapse to a single pane on narrow screens

HR & People Management

Manage employee profiles, org structure, and HR documents. The hierarchy drives the Work Reports team tab and plan assignment.

Org structure

Employees belong to a Department and a Position. The manager_id on Employee creates a self-referential tree:

Employee::manager()       → BelongsTo Employee (direct manager)
Employee::subordinates()  → HasMany Employee (direct reports)

This hierarchy is used by the Work Reports Team tab (managers see their direct reports' submissions) and by work report review authorization.

Employee codes

Employee::nextCode() generates sequential codes in the format EMP-00001. It queries MAX(CAST(SUBSTRING(employee_code,5) AS UNSIGNED)) with withTrashed() so codes are never reused after soft-deletion.

Plans

Full-lifecycle plan management with task tracking, employee assignment, progress calculation, and a client-side view for assigned employees.

Plan lifecycle

Plan categories

daily, weekly, monthly, quarterly, annual, personal, team, project, strategic

Plan numbers

Plan::nextNumber() generates PLN-00001 format codes, querying with withTrashed() to prevent reuse after soft-deletion.

Progress tracking

Plan::updateProgress() recalculates progress as floor(completed_tasks / total_tasks * 100). Called automatically by PlanTaskController@toggle whenever a task is checked or unchecked.

Tags

Stored as a JSON array in plans.tags. The form input accepts a comma-separated string; PlanController::validatePlan() splits it to an array before saving:

$data['tags'] = array_values(array_filter(array_map('trim', explode(',', $data['tags']))));

Employee assignment

Plans use a plan_employees pivot table (BelongsToMany). On create/update: $plan->employees()->sync($employeeIds). The corresponding relationship on Employee is Employee::plans().

Duplicate action

PlanController@duplicate creates a copy of the plan with a new plan_number, status reset to draft, and all tasks copied (statuses reset to pending). Employee assignments are not copied — the duplicate starts unassigned.

Client view (assigned employees)

Routes GET /plans and GET /plans/{plan} are under the standard client auth + active middleware. Access control is enforced in the controller:

  • index — shows only plans where plan_employees.employee_id = auth user's employee
  • show — aborts 403 unless assigned, or the user is admin/editor
  • storeComment — same assignment check before allowing a comment; posts to plans.comments.store (not the admin route)

Admin show page layout

Two-column layout: left 2/3 has progress bar, tasks (with add-task Alpine.js panel), comments, and attachments. Right 1/3 has details sidebar, assigned employees, tags, notes, and a danger zone (delete) at the bottom. The Duplicate and Archive buttons use <form class="contents"> to participate as flex children alongside the Back and Edit links without breaking the flex layout.
